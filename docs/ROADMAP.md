## 基本方針: 実データを最優先で通す

**「実データ読込が最大の山であり、これができてからすべてが始まる」** を中心に据える。
最小描画（ピル＋エッジ）だけで実データは表示できるので、折りたたみ・レーン圧縮・
設定メニューなどの **作り込み UI（full プロトタイプの機能）は最後に回す**。
作り込みは「土台」ではなく「磨き」であり、実データが通ってからやればよい。

潰すべき不確実性を、質の違う順に並べる:

1. **データ**（実セッションから Fork 親子を復元できるか）← 最大の山・最優先
2. **プラットフォーム**（拡張が起動するか）
3. **UI**（綺麗に描けるか・作り込み）← 最後

### 確定済みの設計決定

- **Webview はエディタ領域に開く**（サイドバー/パネルではなく）。
  → API は `vscode.window.createWebviewPanel`（＝**WebviewPanel** 方式、`ViewColumn` 指定）を使う。
  通常のファイルタブと同じ扱いで、分割・移動でき、グラフに必要な横幅を広く取れる。
  `registerWebviewViewProvider`（サイドバーにドックする WebviewView 方式）は**使わない**。
- **エディタ右上（タブ列の右側）にアイコンボタンを置く**。ホバーで「Fork Graph: Open」、
  クリックで上記 Webview をエディタ領域に開く。公式 Claude Code と同じ入口。
  → `package.json` で `menus.editor/title` に `group: "navigation"` でコマンドを登録し、
  そのコマンドに `icon`（SVG）を付ける。実装は数行。必要素材は**アイコン SVG 1枚**。
  （参照: [reference-claude-code/README.md](../reference-claude-code/README.md) §2 / 公式の `editor.openLast` 実装）
- **ノードクリックは公式拡張連携でセッションをエディタ領域に開く**（MVP3）。
  `vscode.commands.executeCommand("claude-vscode.editor.open", sessionId, undefined, vscode.ViewColumn.Active)`
  を呼ぶ。公式 `extension.js` で同じ呼び出しを確認済み。非公開コマンドのため存在確認＋フォールバック必須。
  （詳細は下記 MVP3）

### UI の切り分け（full UI を最後に回すための原則）

| 区分                         | 内容                                                                         | いつ作るか                                       |
| ---------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------ |
| **機能に必要な最小UI** | 最小描画（ピル＋エッジ）／クリックで開く／メモを表示・編集できる素朴なパネル | 各機能のマイルストーンで**都度最小限だけ** |
| **作り込みUI**         | 折りたたみ・レーン圧縮・文字数スライダー・設定メニュー・リッチな右パネル     | **最後にまとめて移植（MVP5）**             |

メモ編集（コア機能）は華やかな右パネルではなく、メモ欄＋ピンボタンだけの素朴な版で先に出し、
豪華版は最後の full UI 移植で差し替える。

- ~~Fork 時は親履歴が**同一 uuid のまま子ファイルにコピー**され、分岐後に追記される。~~
- ~~検出 = **同一プロジェクト内で uuid プレフィックスを共有する2ファイル**。共有が切れる点＝分岐点。~~

**残課題（MVP1 で詰める）**: 親方向ヒューリスティックの厳密化 / `--resume`・`/rewind` の挙動確認

### MVP1.5 — 森（全セッション表示 ＋ セッション間Fork接続）

**背景: Claude Code の Fork は2種類ある**（S0 で両方を確認。詳細は [S0-data-investigation.md](S0-data-investigation.md)）。

| 種別                               | 操作                    | データ上の表れ                                               | 頻度          | MVP対応                              |
| ---------------------------------- | ----------------------- | ------------------------------------------------------------ | ------------- | ------------------------------------ |
| **Type 1: セッション内分岐** | `/rewind`（巻き戻して再送信） | **1 `.jsonl` 内**で同じ `parentUuid` に子が複数    | 多（339中48） | ✅ MVP1 で実装済み（1ファイル→1木） |
| **Type 2: セッション間Fork** | `--fork-session` / VS Code「Fork conversation from here」 | **別 `.jsonl`** が分岐点までのアクティブパスをコピーして分岐（uuid/sessionId は振り直し・timestamp は保存。ただし**末端 1 件のみ再スタンプ**） | 稀（339中1）  | ✅ MVP1.5 で実装済み     |

> ⚠️ 更新（2026-06-08 / [S1-timestamp-merge-design.md](S1-timestamp-merge-design.md)）: Type2 検出は当初「uuid プレフィックス共有」で
> 行う設計だったが、現行 Claude Code（v2.1.163）の Fork は **uuid も sessionId も remap** するため uuid では繋がらない。
> Fork で保存される **`timestamp`（＋type＋内容）を結合キー**に、複製プレフィックスを canonical ノードへ畳む方式へ変更済み。
>
> ✏️ 追記（2026-06-11 / [S1 §10](S1-timestamp-merge-design.md)）: 再調査により (1) Fork がコピーするのは
> ファイル全体ではなく**アクティブな DAG パスのみ**（放棄分岐＝dead-fork entries は残らない）、
> (2) **fork 地点の末端 1 件は timestamp が再スタンプされる**（key で畳まれない既知の限界。
> `promptId` / `requestId` による二次結合の補修案あり）、の 2 点が判明。

**やったこと**:

1. プロジェクトフォルダ（`<encoded-cwd>/`）内の **全 `.jsonl` を読み込む**。各ファイルは MVP1 の
   ローダで木になる（Type 1 込み）。→ この時点で **複数の木＝森**（複数ルート）。
2. 全レコードを `key = timestamp + type + 内容` で canonical 化し（`canonicalize()`）、
   ファイル間で複製された共有プレフィックスを 1 組のノードに畳む。親も key で繋ぐため
   処理順序に依存せず、分岐点から自然に枝分かれする（**Type 2 Fork**）。
3. 森として配置（各木にレーン帯を割り当て、横に並べる）。

**なぜ MVP1 と分けたか**: MVP1 は「1セッションが木になる」ことの確認に絞った。森は
「**全セッションを1枚に並べ、セッション間Forkで繋ぐ**」という別レイヤーで、ここで初めて
**製品の最終的な見た目（プロジェクトの全系譜）**になる。

**実装メモ**:

- `sessionLoader.ts` に `loadProjectAsForest(projectDir)` と `canonicalize()` を実装済み。
  ~~Type 2 検出は S0-doc §4 のアルゴリズム（順序付き uuid リストのプレフィックス共有長）~~
  → uuid では繋がらないため **timestamp＋内容の canonical 化**に変更（[S1 §3](S1-timestamp-merge-design.md)）。
- 所属判定はフォルダ名でなく各レコードの `cwd` を使うと堅い（S0-doc §1）。
- ~~親方向の決定（どちらがオリジナルか）の厳密化~~ → 順序非依存の canonical 化では**そもそも不要**
  （どちらを代表にしても DAG 形状は同じ）。なお向き自体は「共有領域に行き止まりが介在する側がオリジナル」で
  機械的に判定可能と判明（[S1 §10.1](S1-timestamp-merge-design.md)）。
- `--resume` は**同一ファイルへの追記**（別ファイル化しない）ことを確認済み（[JSONL-FORMAT.md](JSONL-FORMAT.md) §5.1）。

**残課題**: なし。fork 地点の末端 1 件（timestamp 再スタンプ）の二次結合（`promptId` / `requestId`）も
✅ 実装済み（2026-06-11。[S1 §10.3](S1-timestamp-merge-design.md)。実データで Fork 7 回ぶんの併合を検証、再デプロイ済み）。

**完了条件**: あるプロジェクトの**全セッションが森として1枚に並び**、`--fork-session` で
分岐したセッション同士が**線で接続されて**表示される。 ✅ 達成済み（2026-06-08。
12 ファイル → 5 ルート木・48 user ノードの森を実データで確認、拡張も再デプロイ済み）

### MVP3 — セッション再開（公式拡張連携・エディタ領域で開く）

> 由来: MVP1 への追加要望（2026-06-05）。「グラフのプロンプトをクリックしたら、
> **公式 Claude Code for VS Code 拡張**でそのセッションを開いてほしい。拡張機能間で連携し、
> 開く形式は**エディタ領域**にしてほしい」。

**やること**: ノードクリック → 公式拡張に該当セッションを**エディタ領域で**開かせる
（外部ターミナル/CLI ではなく、拡張機能間の連携で開く）。
ここで初めて **ノードクリック → 拡張本体へのメッセージ通信**（Webview の `postMessage` →
拡張側 `onDidReceiveMessage`）を最小限だけ実装する（作り込み UI は不要、クリックを拾えれば十分）。

**連携方法（確定・公式 `extension.js` で検証済み / 2026-06-05）**:
公式が内部で使っている非公開コマンドをそのまま呼ぶ。

```js
await vscode.commands.executeCommand(
  "claude-vscode.editor.open", // editor.open = エディタ領域（New Tab）で開く
  sessionId,                   // = 該当 .jsonl のファイル名 UUID
  undefined,                   // initialPrompt（指定しない）
  vscode.ViewColumn.Active     // ← エディタ領域に開く指定
);
```

公式自身が `executeCommand("claude-vscode.editor.open", e.request.sessionId, void 0, ViewColumn.Active)`
と呼んでいるのを確認済み（`editor.open(h,_,b)` → `createPanel(h,_,b)`、`h`=sessionId、`b`=ViewColumn）。
（参照: [reference-claude-code/README.md](../reference-claude-code/README.md) §3）

**重要な含意（粒度はセッション単位）**:
この連携は **`sessionId`（=ファイル単位）で開く**。**1レーン ＝ 1セッション ＝ 1 sessionId** であり、
**同じレーン上のノードはすべて同じ sessionId を共有する**（公式の「Fork conversation from here → 新規プロンプト送信」で
初めて新しい `.jsonl`＝新 sessionId が history に残るため、セッション数 ≒ レーン数）。
よって「どのノードを押しても同じ sessionId」になるのはレーン内では当然の挙動で、欠陥ではない。
クリックで開く価値が十分に出るのは **複数セッションを森として並べたとき**（各レーン＝別 sessionId）なので、
MVP3 と「複数セッション読込（森）」は近い時期にセットで進めるのが自然。

**方針: 地点単位の再開は追わない（過剰機能として明示的に除外）**:
このプロジェクトの本質は「セッション再開がやりづらい」の解消＝**履歴を探しやすくし、アクセスのしやすさを維持する**こと。
メッセージ地点単位で開く（`--resume-session-at <uuid>` 相当）はこの `editor.open` では不可だが、
**そもそも肩代わりする必要がない**。`editor.open(sessionId)` でレーンのセッションをエディタ領域に開いたあと、
ユーザーが公式 UI 側で該当ノードの **Fork conversation from here** を押せば、地点単位の分岐は自分で到達できる。
よって Fork Graph は **sessionId 単位の再開＋履歴の探しやすさ**に専念し、地点単位の再開は実装しない
（CLI `--resume-session-at` への妥協も不要）。これは将来も基本的に追わない方針とする。

**リスク**: `claude-vscode.editor.open` は**非公開コマンド**。①公式拡張がインストール＆有効である前提、
②将来のバージョンで引数仕様が変わりうる。→ 呼ぶ前に
`vscode.extensions.getExtension('anthropic.claude-code')` で存在確認し、
未導入時はフォールバック（CLI `claude --resume <id>` 起動 or 案内）を用意する。

**既知の制約（2026-06-11 判明）**: 公式 UI の履歴で「削除」したセッションは、ファイルは残るが
VS Code globalState の `hiddenSessionIds` に登録され、`editor.open` に渡しても
**エラーなく新規チャットにすり替わる**（公式拡張がセッション読込失敗時に resume を黙って捨てる仕様）。
データ自体は無傷で、CLI `claude --resume <id>` なら開ける（実機検証済み）。
→ 対応は MVP5 の右パネルで行う（下記「削除済み — CLI で開く」）。

**完了条件**: グラフのノードをクリックすると、公式 Claude Code がそのセッションをエディタ領域に開く。
これで「探す→開く」の実用ループが一周する。 ✅ 達成済み（2026-06-08）

**実装済みの内容**:
- ノード(`session` 持ち)に `class="node clickable"` ＋ ホバー強調 ＋ クリックで
  `vscode.postMessage({type:'openSession', sessionId})`。
- 拡張側 `panel.webview.onDidReceiveMessage` → `openSession(sessionId)`:
  `getExtension('anthropic.claude-code')` で存在確認 → `executeCommand('claude-vscode.editor.open',
  sessionId, undefined, ViewColumn.Active)`。
- フォールバック: 公式拡張が無ければ警告＋「ターミナルで開く」で `claude --resume <id>` を起動。
- ノードモデルに `session`（= `rec.sessionId`）を追加済み（`sessionLoader.ts`）。

### MVP6 — 仕上げ・差別化機能

> ✏️ 改定（2026-06-12）: 旧リスト（PROTOTYPES.md §6.3 ベース）を再編。
> 「ホバーでプロンプト全文ツールチップ」は MVP5 で実装済みのため削除。
> 「エッジ重なり修正」「バージョン互換性考慮」は対象外（必要になったら再検討）へ。

**継続項目**:

1. **★ピン付きだけ表示フィルタ**（☰メニューに増設）
   - ON にすると**ピン付きノード＋その祖先パスだけを表示**（木構造は維持）。
     ピンの無いノードは非表示にし、レーン圧縮も適用して「自分の重要地点だけのコンパクトな地図」にする。
2. **検索ボックス**（メニューバーに増設）
   - **プロンプト＋メモを横断**してインクリメンタル検索。ヒット以外のノードを薄暗くする
     （ヒットを消さないのは、位置＝系譜の文脈ごと見つけたいため）。
3. **「いつ」の表示**（右パネル）
   - 選択ノードの timestamp を右パネルに表示（絶対時刻＋「3日前」等の相対表記）。
   - 実装メモ: 現状の `GraphNode` は timestamp を持っていないため、`sessionLoader.ts` でフィールド追加が必要。

**追加項目（2026-06-12 決定）**:

4. **フォルダ選択機能**（☰メニューに増設）
   - `~/.claude/projects/` 配下のうち **`.jsonl` を持つフォルダだけ**を列挙し、
     **最終更新が新しい順**に並べる。選択するとそのプロジェクトの森に表示を切り替える。
   - 現状の「ワークスペース対応フォルダ→無ければ直近フォルダ」の自動選択は初期値として残す。
5. **Fork Graph 画面だけのソフトデリート**
   - 右パネルに「ソフトデリート」操作: **選択ノード自身＋それ以降（子孫）**をグラフから非表示にする。
   - あくまで **Fork Graph 内だけの表示制御**。`.jsonl` にも公式の `hiddenSessionIds` にも一切触れない
     （公式の「削除」とは独立。データ破壊リスクゼロ）。
   - 保存は `globalState`（メモと同じ**安定キー**方式。再起動後も維持）。
   - ☰メニューに「**非表示済みチャットから復元**」一覧を増設し、選んで個別に元へ戻せる。
6. **自動リロード（常に最新状態を表示）**
   - ユーザープロンプトが送られる等でセッションファイルが伸びたら、自動で再読込・再描画する。
   - **方式: `FileSystemWatcher`** で表示中プロジェクトの `*.jsonl` を監視し、
     変更イベントを debounce（数百 ms）してからリロード。
   - 方式の選定理由: 拡張内で完結し**配布が容易**（ユーザー側の Hooks 設定が不要）。
     CLI・パネル・ツール実行などどの経路の追記でも拾える。トークン消費は無し
     （なお Hooks 連携でもトークンは消費しない。Hooks の利点は「プロンプト送信時だけ発火する精密さ」だが、
     ユーザーごとの settings.json 設定が必要になり配布に不向きなため不採用。Agent SDK は不要）。
   - リロード時も **UI 状態（選択・折りたたみ・検索・スクロール）を安定キーで維持**し、データだけ差し替える。

**実装済み（2026-06-12）**: 6 機能すべて実装・デプロイ済み。
Webview を「初期データ埋め込み＋ `postMessage('data')` で差し替え可能」な構造に再編し、
UI 状態（選択・折りたたみ・ソフトデリート）を**安定キー**で保持してリロード耐性を確保。
自動リロードは `fs.watch` ＋ 600ms debounce（メモ入力中は blur まで適用を退避）。
フォルダ選択のラベルは最新 `.jsonl` 先頭レコードの `cwd` から取得（encode 名は復号不能のため）。
詳細: [mvp6-polish-features.md](../fork-graph-ext/docs/implementation/mvp6-polish-features.md)

**完了条件**: ピンフィルタ・検索・時刻表示・フォルダ切替・ソフトデリート/復元・自動リロードの
6 機能が実データ上で動き、リロードしても UI 状態と永続データ（メモ・ピン・ソフトデリート）が壊れない。

## 開発の回し方（最新コードの反映手順）

> npm 系コマンドは**すべて `fork-graph-ext/` の中**で実行する（`package.json` はそこにある）。
> プロジェクト直下 `fork-graph/` で叩くと `ENOENT: package.json not found` になる。

### A. 開発中（速い・日常）— F5 デバッグ

1. `fork-graph-ext` で **`npm run watch`** を起動しっぱなし（保存ごと自動コンパイル）
2. **F5** で開発ホスト窓を起動
3. コード編集 → 保存（watch が `out/extension.js` を更新）
4. **開発ホスト窓で `Ctrl+R`**（この窓だけ効く）→ 反映
   - Webview の中身を変えたら、反映後に **Fork Graph タブを閉じて開き直す**

### B. 節目の本番確認 — インストール版（.vsix）に反映 ★主手順

```bash
cd ~/Desktop/claude-app/fork-graph/fork-graph-ext   
npm run redeploy
```

`redeploy` = `compile && vsce package --allow-missing-repository && code --install-extension fork-graph-0.0.1.vsix`
（コンパイル → .vsix 再生成 → 再インストールを1発で実行）

そのあと **通常窓**で反映:

- `Ctrl+Shift+P` → **Developer: Reload Window**（**`Ctrl+R` は通常窓では効かない**＝Open Recent に割当）
- `Ctrl+Shift+P` → **Fork Graph: Open**（またはエディタ右上アイコン）

#### ハマりどころ（既知）

| 症状                                      | 原因 / 対処                                                                                     |
| ----------------------------------------- | ----------------------------------------------------------------------------------------------- |
| `ENOENT: package.json`                  | `fork-graph-ext` の外で npm を実行している。`cd fork-graph-ext` してから                    |
| `Do you want to continue? [y/N]` で停止 | LICENSE 警告のプロンプト。`LICENSE.txt` を置いたので**現在は出ない**。出たら `y`      |
| `Ctrl+R` でリロードされない             | 通常窓では `Ctrl+R`=Open Recent。**コマンドパレットの Developer: Reload Window** を使う |
