# S1 — timestamp 結合によるファイル間 Fork 統合の再設計

> S0（[S0-data-investigation.md](S0-data-investigation.md)）で確定したはずの「Fork は同一 uuid のまま履歴をコピーする」という前提が、
> 現行 Claude Code（v2.1.163）では**成り立たないこと**が判明した。本ドキュメントはその根拠と、
> uuid に代わる結合キー **`timestamp`** を用いた再設計アルゴリズムを記録する。
>
> 対象コード: [fork-graph-ext/src/sessionLoader.ts](../fork-graph-ext/src/sessionLoader.ts)
> 関連: [ROADMAP.md](ROADMAP.md)（MVP1.5 森） / [JSONL-FORMAT.md](JSONL-FORMAT.md)（レコード構造）
>
> 作成日: 2026-06-08 / 更新: 2026-06-11（§10 追補: Fork は「アクティブパスのみ」コピーし、**fork 地点の末端 1 件は timestamp が再スタンプされる**ことが判明。関連箇所を訂正）

---

## 1. 背景 — 何が問題だったか

### 1.1 旧前提（S0 §3）

MVP1.5「森」は、次の 1 点に全面的に依存していた。

> Fork（`--fork-session` / `/branch`）時は、親履歴が**同一 uuid のまま**子ファイルにコピーされる。
> よって全ファイルを **uuid で重複排除**すれば、別ファイルへの Fork も「分岐点 uuid に子が複数」という
> Type1（セッション内分岐）と同じ形に畳まれ、森が建つ。

実装は [sessionLoader.ts:151](../fork-graph-ext/src/sessionLoader.ts#L151) の `if (d.uuid && !recs.has(d.uuid))` で重複 uuid を捨てることでこれを実現していた。

### 1.2 旧前提は現行版で無効

**公式仕様（Claude Code Docs / Session Storage、context7 経由で確認）**:

> The `forkSession` function creates a new session by reading source entries,
> **rewriting session IDs, and remapping message UUIDs**. It then appends these
> transformed entries under a new key.

つまり Fork は履歴コピー時に **`sessionId` も `uuid` も新規に振り直す**。
`--continue` / `--resume`（同一セッション継続・uuid 保持）とは別物。

### 1.3 実データによる裏付け（v2.1.163 / 本プロジェクトの 10 セッション）

| 計測 | 結果 | 含意 |
| --- | --- | --- |
| 各レコードの `sessionId` | **必ず自分のファイル名 UUID と一致** | Fork 時に振り直されている |
| ファイル間で共有される `uuid` | **0 個** | uuid での結合は原理的に不可能 → 重複排除が一度も発動しない |
| ファイル間で共有される `timestamp` | **388 個**（例: `715448d0` ↔ `d8b477e8` で 346 個） | **timestamp は保存される** = 新しい結合キー |
| 10 ファイル中の先頭プロンプト | **7 つが同一** | 同じ会話が 7 回複製されている |

### 1.4 これが生む 2 つのバグ

1. **親子でないのに瓜二つのノード**: Fork が全履歴を新 uuid で複製 → 重複排除が効かず、同じ会話プレフィックスが
   **N 本の独立した木**として並ぶ。DAG 上は別 uuid なので親子関係を持たない。
2. **表示と違うチャットが開く / 別レーンが同じチャット**: 各ノードの `session` は自分のファイルの sessionId
   （[sessionLoader.ts:257](../fork-graph-ext/src/sessionLoader.ts#L257)）。複製ノードは「元会話の一部」に見えても実体は
   どれかの Fork ファイル所属 → クリックで別セッションが開く。

> 結論: **アルゴリズムのバグではなく、データモデルの前提が現行版で無効になった**。
> 1 ファイル内のプロンプト認識（②③④）は健全。欠陥は**ファイル間を繋ぐ結合キーの選択**のみ。

## 2. 設計原則 — timestamp + 内容を結合キーにする

Fork で生き残る不変量は **`timestamp`（ミリ秒・UTC）と発話内容**だけ。これを正規化キーにして、
複数ファイルに複製された「同じ論理イベント」を **1 つの正規ノード（canonical node）に畳む**。

| 不変量 | Fork 時 | プロンプト手動流用時（別会話で同じ初手を再利用） |
| --- | --- | --- |
| `uuid` / `sessionId` / `parentUuid` | 振り直される（使えない） | 当然別 |
| **`timestamp` + 内容** | **完全一致が連続して並ぶ** | **新規送信なので一致しない** |

**この区別が決定的**: 手動流用は新しい timestamp になるため、長い連続区間でミリ秒一致する真の Fork とは
確実に区別され、別の木として正しく分離される（ユーザー要件）。

---

## 3. アルゴリズム（順序非依存・キーで親も繋ぐ）

> ⚠️ **改訂（2026-06-08 実装時）**: 初版は「mtime 最古のファイル＝オリジナル＝代表」とし、処理順序に依存していた。
> しかし **mtime（最終更新）は作成順を表さない**（長く使われた幹は最後まで追記されて mtime が最新になり、
> 早期に放棄された小さな fork のほうが mtime が古くなる）。実データで幹 `715448d0`（702 レコード）より、
> idx 18 で分岐した 47 レコードの fork `2589b6fc` のほうが mtime が古く、後者が代表に選ばれて
> **幹を誤って継ぎ接ぎ**していた。下記の通り **処理順序に一切依存しない方式**へ改めた。

```
①'  全 .jsonl の全レコードを保持（dedup しない）
①'' 各レコードを key=(timestamp+type+内容) 化し、「親 = 親レコードの key」で DAG を構築
       └ Fork コピーは親も一緒に複製され timestamp を保つので「親の key」も全ファイルで不変
       └ よって mtime 等の処理順序に依存せず、同一 key は 1 ノードに畳まれ枝も正しく付く
②③④ 既存のまま（user 抽出 → nearestUserParent → レーン割当）
```

### ①' 全レコード保持

全 `.jsonl` を読み、uuid を持つ全レコードを保持（**重複排除しない**）。各レコードに由来 sessionId が付いている。

### ①'' canonical 化（新規の核心）

**(a) 各レコードに正規キーを与える**

```
key(rec) = rec.timestamp + "|" + rec.type + "|" + fingerprint(rec)
```

- `fingerprint` = `message.content` を安定文字列化したもの（user/assistant）、`attachment` 本体、`system` の subtype 等。
- **根拠**: timestamp 単独だと同一ミリ秒の別イベント衝突が起こり得る。type と内容を混ぜることで、
  Fork 複製（3 つとも一致）だけが畳まれ、偶然の別物は畳まれない。

**(b) 「親も key で繋ぐ」（順序非依存の要）**

各ファイル内で `parentUuid` を解決して親レコードを取り、その親の key を覚える。

```
parentKey(rec) = key( 同ファイル内の parentUuid が指すレコード )   // 親が無ければ null
```

- **根拠（決定的）**: Fork は分岐点までの **DAG パスを親ごと一緒に複製**し timestamp を保つので、
  ある論理イベントの `key` も「その親の `key`」も**どのファイルでも同一**。よって
  `key → parentKey` を集めるだけで、**処理順序やどのファイルが代表かに一切依存せず**正しい DAG が建つ。
  代表（その key の uuid）は最初に見た 1 個体を使えば十分（どれを選んでも DAG 形状は不変）。
  - 実測（2026-06-11 / 14 ファイル）: 共有キー 464 個すべてで親キーの不一致 **0 件** → 不変性は成立。
  - ⚠️ ただし複製されるのは「ファイル全体のプレフィックス」ではなく**アクティブパスのみ**で、
    **fork 地点の末端 1 件は timestamp が再スタンプされる**例外がある（§10）。パス上の祖先は必ず複製されるため
    parentKey 不変性自体は壊れないが、末端 1 件は key 不一致で畳まれず重複ノードになる。

**(c) canonicalId と親の翻訳**

```
canonId(key)        = 代表レコードの uuid
node(key).parent    = canonId( parentKey(key) )
```

同一 key の全コピーがこの 1 ノードに畳まれ、分岐点では「同じ親 key を持つ別 key の子」が複数ぶら下がる
＝ Type1（/rewind）も Type2（--fork-session）も同じ「分岐点に子が複数」の形になる。

**(d) 偶然衝突の扱い**

同一 key が**異なる親 key**を持つことは、無関係レコードの偶然一致（同一 ms＋同一内容＋同一 type）でしか起きず極めて稀。
その場合は**最初に確定した親を採用**（先勝ち）して安全側に倒す。

### ②③④ 既存ロジックそのまま

canonical 化済み（uuid=canonicalId、parentUuid=canonical 親）のレコード集合を、
`buildGraphFromRecords`（[sessionLoader.ts](../fork-graph-ext/src/sessionLoader.ts)）にそのまま流す。

- **② user 抽出**（`isOnlyToolResult` / `isHarnessNoise`）: 変更不要。
- **③ nearestUserParent**: 変更不要。parentUuid が canonical なので**ファイル境界を越えて** user 祖先に届く。
- **④ レーン割当（原則 A/B）**: 変更不要。`order` は **timestamp 昇順**（信頼できる唯一の時系列）で渡し、枝の並びを安定させる。

## 4. ノードの sessionId（クリック先）の決定

canonical ノードは複数セッションに現れるため、`session` を 1 つに決める。各 key に「出現セッション集合」を持たせ、選ぶ。

| 方針 | 実装 | 性質 |
| --- | --- | --- |
| ~~最古オリジナル~~ | ~~代表（最古ファイル）の sessionId~~ | **不採用**: mtime が作成順を表さず、放棄 fork を誤って開く |
| **最長継続セッション（採用）** | そのノードを含むセッションのうち**レコード数が最大**のものを選ぶ | 共有プレフィックスは**幹（本流）**を開く。正確かつ「続きから作業」に直結 |
| 全セッション選択 | 出現集合を UI 提示 | 正確だが一手間（将来オプション） |

- **採用理由**: 分岐後の固有ノードは 1 セッションにしか現れないので、そこは当然そのセッションを開く。
  曖昧なのは共有プレフィックスのみで、そこは**最も長く続いた幹**を開くのが最有用（例: 共有の「すべて5ファイル」は
  47 レコードの放棄 fork ではなく 702 レコードの本流 `715448d0` を開く）。出現セッション集合は将来の UI 用に内部保持。
- **補強（2026-06-11）**: 実データでは本流 `715448d0`（702 件）自体が `d8b477e8`（538 件）からの **Fork** だった
  （§10.1）。つまり「オリジナル＝本流」は成り立たず、Fork 側が事実上の幹になることが普通にある。
  「オリジナル優先」ではなく「最長継続優先」を採った本方針は、この点でも正しかった。

## 5. 実データでの動作例（`715448d0` ↔ `d8b477e8`、共有キー 368 個 / 共有 timestamp 346 個）

> ✏️ 改訂（2026-06-11）: 初版は「古い方が代表になる」という mtime 前提の説明で、Fork の向きも未確定だった。
> 実測で向きが確定したため書き直し（詳細な根拠は §10.1）。

- **`d8b477e8` がオリジナル**（538 件。共有領域の中に、Fork へコピーされなかった行き止まり分岐＝dead-fork entries が 91 件介在）。
- **`715448d0` がその Fork**（702 件。先頭 368 件が連続プレフィックスとしてコピーされ、その直後に
  timestamp 再スタンプ済みの末端コピー 1 件、以降が固有の続き）。Fork 後も長く使われ、事実上の本流になった。
- canonicalize の動き:
  1. 両ファイルの共有 368 キーが 1 つの canonical ノード列に畳まれる（代表はどちらの個体でもよい。DAG 形状は不変）。
  2. `d8b477e8` 固有の行き止まり 91 件・分岐後の続き、`715448d0` 固有の続きは、それぞれ共有部から**枝として接続**。
  3. 結果: 瓜二つに増殖していた木が**共有部 1 本＋各 Fork の枝**という git-graph 状の 1 つの森に統合。
     クリックも各ノードの正しいセッションへ。
- ⚠️ 例外: `715448d0` の 369 件目（fork 地点の末端コピー）は timestamp が再スタンプされているため key が一致せず、
  オリジナル側の同一発話と**畳まれない**（§10.2）。現状は非 user レコードかノイズ除去対象に当たるため表示上は無害。

## 6. エッジケースと扱い

| ケース | 挙動 | 根拠 |
| --- | --- | --- |
| プロンプト手動流用（別会話で同じ初手） | 別 timestamp → 別 key → **畳まれず別の木** | §2。新規送信は新しい timestamp |
| 同一 ms の別イベント衝突 | type+内容も鍵に含むため**誤統合しない** | §3(a) |
| 孤立した 1 件の偶然一致 | 同一 key・異なる親 key は先勝ちで安全側に倒す | §3(d) |
| 古いセッションから後日 Fork | 順序非依存の key 結合なので、ファイルの新旧・mtime に関係なく正しく畳まれる | §3(b)（✏️ 旧記述「mtime 昇順＝オリジナル優先」は廃止済みの初版方式。訂正） |
| 部分的にだけ共有して分岐 | per-record キー一致＋parent 翻訳で、共有分だけ畳まれ残りは枝 | 明示的な最長共通接頭辞計算は不要 |
| オリジナル側の行き止まり分岐（dead-fork entries） | Fork にコピーされず、オリジナルだけの枝として残る（畳む必要なし） | §10.1。Fork は**アクティブパスのみ**コピーする |
| **fork 地点の末端 1 件（timestamp 再スタンプ）** | 一次 key では畳まれないが、**安定 ID による二次結合で併合**（✅ 実装済み） | §10.2 / §10.3。`promptId`（user）・`requestId`（assistant）が一致条件 |
| timestamp 欠落 / uuid 無しのメタレコード | そもそもノード化対象外（②で除外済み） | [JSONL-FORMAT.md](JSONL-FORMAT.md) §3 |

## 10. 追補（2026-06-11）— 再調査で判明した Fork 複製の正確な挙動

> 「チャットの同定アルゴリズムに不備がないか」の再調査結果。全 14 ファイルの実測と、
> 公式リポジトリ CHANGELOG（context7 経由）の **dead-fork entries** という用語の確認に基づく。
> 結論: **§3 の骨格（key＋parentKey 結合）は健全**だが、§2 の「Fork は timestamp を保存する」という
> 前提には**例外が 1 つ**あり、関連する説明（複製範囲・Fork の向き）に誤りがあった。

### 10.1 Fork がコピーするのは「ファイルのプレフィックス」ではなく「アクティブな DAG パス」

- オリジナル `d8b477e8` の共有領域には、Fork `715448d0` に**コピーされなかったレコードが 91 件**介在する
  （hook 出力 30 件・tool_result 21 件・assistant 33 件など）。91 件すべてが**共有子孫を持たない行き止まり**
  （`/rewind` 等で放棄された分岐）だった。公式 CHANGELOG もこれを "dead-fork entries" と呼び、
  `--resume` が読み飛ばす対象として言及している。
- 対して Fork ファイル側 5 つ（`4e3222a3` `5817c3ad` `5e619432` `6e190b3b` `715448d0`）は、
  共有レコードが**例外なく連続した先頭プレフィックス**を成す（穴 0）。
- **Fork の向きはここから機械的に判定できる**: 共有領域に行き止まりが介在する側がオリジナル、
  連続プレフィックスの側が Fork。これにより `715448d0`（幹・702 件）は `d8b477e8` の Fork と確定
  （従来の調査メモは向きを逆に推定していた）。
- アルゴリズムへの影響: **なし**（むしろ好都合）。パス上の祖先は必ず一緒にコピーされるため、
  §3(b) の parentKey 不変性は成立する（実測: 共有 464 キーで親キー不一致 0 件）。

### 10.2 例外: fork 地点の末端 1 件は timestamp が再スタンプされる

- 全 Fork ファイルで、コピーされたプレフィックスの**直後の 1 件**が「オリジナルのある発話と
  内容完全一致なのに timestamp だけ異なる」レコードだった（5/5 ファイルで再現）。
- 決定的証拠（`715448d0` の例）: オリジナル `d8b477e8` の最終 assistant 発話（03:34:54.676Z）と
  `requestId`（`req_011CbjQoXZPFNRgMp8DbWaes`）・`message.id`・`usage`・本文がすべて一致する逐語コピーが、
  Fork 作成時刻（08:36:24.467Z）の timestamp で書かれている。API を再呼び出しせずにこの一致はあり得ない
  ＝**コピー時に末端 1 件だけ timestamp が書き換えられている**。
- 帰結: この末端 1 件は key が一致せず canonical に**畳まれない**。canonical DAG 上は
  「同じ親を持つ内容同一の兄弟」として重複する。
  - 現状の実害は小さい: 末端が assistant / tool_result / `[Request interrupted…]` 型の user であれば
    user ノード化されない（②のノイズ除去で消える）ため、グラフには現れない。
  - ただし**人間の user 発話を fork 地点にして Fork した場合**は、同じプロンプトのノードが
    幹と枝に 1 つずつ並ぶ「瓜二つノード」が小規模に再発し得る。→ §10.3 の二次結合で解消（✅ 実装済み）。

### 10.3 補修案 — Fork を生き残る安定 ID による二次結合

timestamp が再スタンプされても、次の ID はコピーでそのまま保存されることを実測で確認した。

| レコード型 | 安定 ID | 実測 |
| --- | --- | --- |
| `user` | `promptId` | 再スタンプされた末端でも 3 ファイル間で同一値（`e881f90e-…`）を確認 |
| `assistant` | `requestId` / `message.id` | 再スタンプされた末端でも一致（上記の決定的証拠そのもの） |

- 補修: ①'' の後に二次パスを足し、「**親 key が同一・型と内容が一致・安定 ID が一致**・timestamp のみ不一致」の
  ペアを同一 canonical に併合する。誤併合リスクは安定 ID の一致が条件なので実質ゼロ。
- ✅ **実装済み（2026-06-11）**: `canonicalize()` に二次結合を実装（代表は timestamp 最古＝オリジナルの真の時刻）。
  実データ検証: 既知の再スタンプペアが 1 ノードに併合・dangling 親 0・同一内容兄弟の残存 0・
  併合数 7 ＝ Fork 系図（関連 9 ファイル − オリジナル 2）の Fork 回数と完全一致。
  手順書: [fork-graph-ext/docs/implementation/stable-id-secondary-merge.md](../fork-graph-ext/docs/implementation/stable-id-secondary-merge.md) /
  検証: [fork-graph-ext/scripts/verify-secondary-merge.js](../fork-graph-ext/scripts/verify-secondary-merge.js)
- 注意: `message.id` は 1 回の API 応答が複数行（thinking / text / tool_use）に分かれても共通なので、
  単独では per-record キーにならない（内容 fingerprint との併用が必須）。timestamp も同一 ms に複数レコードが
  普通に存在する（thinking と text の分割書き込み、並列ツールの同時拒否など。実測で同一 ms ペア多数）ため、
  **「timestamp 単独キー」への単純化は不可**という §3(a) の設計判断は正しい。
