# ブロック単位のテキスト抽出（合体プロンプトの取りこぼし修正）

> 対象コード: [src/sessionLoader.ts](../../src/sessionLoader.ts) の `genuineTextBlocks()` / `fingerprint()`
> 関連: [JSONL-FORMAT.md §5.4](../../../docs/JSONL-FORMAT.md)

## 概要

1 つの `user` レコードの `message.content` には、`<ide_opened_file>` 等の**自動添付ブロック**と
**人の発話ブロック**が同居することがある（例: `[ <ide_opened_file>…, "うまくいきました。一気に" ]`）。

旧実装は「最初の text ブロック」だけを取り出してレコード単位でノイズ判定していたため、
先頭が自動添付だと**本物のプロンプトごとレコードを捨てて**いた。これにより
「全体ロードマップについて…」「MVP1 — 実データ読み込みに進みます」など実プロンプト多数が
グラフから欠落していた（主レーン 17 → 修正後 26 ノードに回復）。

さらに、CLI の resume は履歴を**再シリアライズして追記**することがあり（新 uuid・timestamp 保持）、
その際に合体レコードを**個別レコードに分割**する。同一発話が「合体版」「分割版」の両形式で
存在し得るため、指紋も発話本文基準に正規化しないと重複ノードが生まれる。

## 実装手順

### 1. `extractText` を `genuineTextBlocks` に置き換える
- 目的: 自動添付と同居していても発話本文を救う
- やること: content の全 text ブロックを走査し、`isHarnessNoise` に当たらないものだけを配列で返す。
  string 形式の旧 content にも対応

### 2. ノード抽出条件を「発話ブロックが 1 つ以上あるか」に変更
- 目的: レコード単位のノイズ判定をやめる
- やること: `genuineTextBlocks(d).length === 0` ならスキップ。`prompt` は発話ブロックの連結

### 3. user レコードの指紋を発話本文基準に正規化
- 目的: 合体版と分割版（resume の再シリアライズで生じる）を同じ canonical ノードに畳む
- やること: `fingerprint()` で user かつ発話ブロックがあれば `join("\n")` を指紋にする。
  発話ブロックを持たないレコード（tool_result・純添付）は従来どおり content 全体の JSON

## 判断理由

- **timestamp＋type は鍵に残る**ので、本文だけの指紋でも誤統合は起きない
  （同一ミリ秒・同一本文・user 同士なら同一の論理イベントとみなしてよい）。
- tool_result の指紋を本文化しない理由: 本文を持たず空指紋になると、
  同一 ms の並列 tool_result 同士が衝突するため。

## 検証（2026-06-11 実データ）

- 主レーン 17 → 26 ノード（欠落していた実プロンプト 9 件が正しい位置に直列で復元）
- 再シリアライズ起因の 1 ノード枝 12 本が幹に吸収され、枝は本物の分岐 4 本のみに
- 二次結合チェック（既知ペア併合・dangling 0・同一兄弟 0）も引き続き合格
